Mobile App (Actor)

Description: Native iOS/Android application that consumes the REST API. Handles authentication tokens locally.

#TitleTypeSeverity StatusScoreDescriptionMitigations
9 Token Storage in Insecure Location Information disclosure High Open 7.1 Access and refresh tokens stored in insecure shared preferences or unprotected files on rooted/jailbroken devices can be extracted by malicious apps. Use iOS Keychain and Android Keystore for token storage. Detect rooted/jailbroken devices and limit functionality. Implement certificate pinning.
10 Fake Mobile App Impersonation Spoofing High N/A 6.8 A cloned or repackaged version of the mobile app could be published on third-party stores to steal credentials. N/A — Platform enforces app attestation (App Attest / Play Integrity). Mitigated at the OS level.

API Gateway (Process)

Description: Central ingress point for all REST API traffic. Performs JWT validation, rate limiting, WAF filtering, and routing to downstream services.

#TitleTypeSeverity StatusScoreDescriptionMitigations
11 JWT Algorithm Confusion Attack Elevation of privilege Critical Open 9.0 If the API Gateway accepts 'alg: none' or allows RS256 tokens verified as HS256 using the public key as HMAC secret, attackers can forge arbitrary tokens. Explicitly whitelist allowed JWT algorithms (RS256 only). Reject tokens with 'alg: none'. Use a well-maintained JWT library with algorithm pinning.
12 API Rate Limit Bypass Denial of service High Open 7.5 Distributed clients rotating IPs can bypass per-IP rate limits, overwhelming backend services with excessive requests. Implement rate limiting by user ID (JWT sub claim) in addition to IP. Use token bucket algorithm. Integrate with a threat intelligence feed for known bot IPs.
13 Verbose Error Responses Leak Internal Details Information disclosure Medium Mitigated 4.0 Stack traces or internal service names returned in 500 error responses reveal architecture details useful for targeted attacks. Gateway now returns generic error messages. Internal errors are logged with correlation IDs only. No stack traces in production responses.

Order Service (Process)

Description: Manages order lifecycle: creation, status updates, fulfilment triggers. Writes to the Orders DB and emits events to the message queue.

#TitleTypeSeverity StatusScoreDescriptionMitigations
14 IDOR on Order Retrieval Elevation of privilege Critical Open 8.5 Without per-resource authorization checks, a customer can access another customer's order by manipulating the orderId parameter in the URL. Enforce resource-level authorization: verify JWT sub claim matches the order's owner on every read/write operation. Add integration tests for IDOR scenarios.
15 Order Tampering via Replay Attack Tampering High Open 7.8 An attacker who captures a valid 'create order' request can replay it to create duplicate orders, potentially abusing promotional codes or stock reserves. Implement idempotency keys for order creation. Validate idempotency key uniqueness per user. Expire idempotency keys after 24 hours.

Product Catalog Service (Process)

Description: Read-heavy service providing product listings, search, and detail pages. Backed by an Elasticsearch cluster and PostgreSQL.

#TitleTypeSeverity StatusScoreDescriptionMitigations
16 NoSQL Injection in Search Query Tampering High Open 8.1 Unsanitized search query parameters passed to Elasticsearch queries can be manipulated to exfiltrate hidden product data or cause denial-of-service via expensive aggregation queries. Use parameterized Elasticsearch query DSL. Whitelist allowed query fields. Set query complexity limits and circuit breakers on the Elasticsearch cluster.
17 Price Data Tampering via Cache Poisoning Tampering Medium Open 5.5 If the catalog cache (Redis) can be poisoned with incorrect price data, customers may see wrong prices, causing financial or reputational harm. Sign cached price data with an HMAC key. Validate signatures on cache reads. Implement cache integrity monitoring alerts.

Orders DB (PostgreSQL) (Store)

Description: Relational database storing order records, line items, status history, and customer references. Contains PII.

#TitleTypeSeverity StatusScoreDescriptionMitigations
18 SQL Injection via Order Service Tampering Critical Open 9.3 Unsanitized order fields forwarded to raw SQL queries could allow attackers to dump the entire orders database or modify order statuses. Use parameterized queries or an ORM exclusively. Enable pg_audit for query logging. Apply principle of least privilege: Order Service DB user has no DDL permissions.
19 PII Exposure via Unencrypted Backup Information disclosure High Open 7.0 Database backups stored without encryption could expose customer PII (name, address, order history) if backup storage is breached. Encrypt all backups at rest using AES-256. Store backup encryption keys in a dedicated HSM or KMS. Restrict backup bucket access to backup service role only.
20 No Database Audit Log Repudiation Medium Mitigated 4.5 Without an audit log, changes to order records cannot be attributed to specific users or services, hampering incident response. pg_audit extension is now enabled. All INSERT, UPDATE, DELETE on orders table are logged to a write-once audit log store.

Product Cache (Redis) (Store)

Description: In-memory cache storing product listings and prices. No persistent data; populated from PostgreSQL on cache miss.

#TitleTypeSeverity StatusScoreDescriptionMitigations
21 Unauthenticated Redis Access Information disclosure High Open 8.0 If Redis is not configured with authentication (requirepass) and network access is not restricted, any host in the VPC can read or flush the cache. Enable Redis AUTH. Restrict network access via security groups to Order Service and Catalog Service IP ranges only. Enable Redis TLS.

REST API (HTTPS) (Data Flow)

Description: Mobile app communicates with backend via REST over HTTPS with Bearer token authentication.

#TitleTypeSeverity StatusScoreDescriptionMitigations
22 Missing Certificate Pinning Tampering Medium Open 5.3 Without certificate pinning in the mobile app, a compromised CA or local proxy can perform a MITM attack and inspect or modify API traffic. Implement certificate pinning using the leaf certificate or public key hash. Provide an update mechanism to rotate pinned certificates before expiry.

Customer Browser (Actor)

Description: End-user browser interacting with the storefront. Runs untrusted client-side JavaScript.

#TitleTypeSeverity StatusScoreDescriptionMitigations
1 Session Token Theft via XSS Spoofing Critical Open 8.8 Malicious scripts injected into the page can steal session cookies or OAuth tokens stored in localStorage, allowing an attacker to impersonate the customer. Set HttpOnly and Secure flags on all session cookies. Store tokens in memory only. Implement a strict Content Security Policy (CSP). Sanitize all user-generated content rendered in the DOM.
2 Lack of Client-Side Audit Trail Repudiation Medium Mitigated 4.3 Without server-side logging of significant customer actions (add to cart, checkout initiation), a customer could deny having placed an order. Log all order-critical events server-side with timestamps and IP addresses. Use signed audit log entries.

External IdP (OAuth2) (Actor) — Out of Scope

Reason for out of scope: Managed by a third-party vendor; security posture governed by SLA and vendor compliance certifications (SOC 2, ISO 27001).

Description: Third-party identity provider (e.g. Google, Auth0) used for SSO. Out of direct control of the platform team.

#TitleTypeSeverity StatusScoreDescriptionMitigations
3 Compromised IdP Issues Forged Tokens Spoofing Critical Open 9.1 If the external IdP is compromised or misconfigured, it may issue JWTs for arbitrary users, granting them full access to the platform. Validate JWT signatures against the IdP's published JWKS endpoint. Enforce audience (aud) and issuer (iss) claims. Implement token binding where possible.

Web Frontend (SPA) (Process)

Description: React single-page application served via CDN. Handles rendering, routing, and OAuth2 PKCE flow for authentication.

#TitleTypeSeverity StatusScoreDescriptionMitigations
4 Open Redirect in OAuth Callback Spoofing High Open 7.4 The OAuth2 redirect_uri parameter is not strictly validated, allowing an attacker to redirect the authorization code to an attacker-controlled endpoint. Strictly whitelist allowed redirect_uri values. Use state parameter with CSRF token. Validate redirect_uri on both client and authorization server.
5 Denial of Service via CDN Cache Poisoning Denial of service Medium Open 5.9 Attackers may poison the CDN cache with malformed or malicious responses, causing legitimate users to receive corrupted pages. Use cache keys that include all relevant headers. Validate responses before caching. Implement cache purge procedures.
6 Sensitive Data in Browser Local Storage Information disclosure High Mitigated 6.5 Access tokens or PII stored in localStorage are accessible to any script running on the page, enabling theft via XSS. Tokens are now stored in memory only (React state/context). Sensitive operations require re-authentication. HttpOnly cookies used for refresh tokens.

HTTPS Page Request (Data Flow)

Description: Customer browser requests SPA assets and API data over HTTPS.

#TitleTypeSeverity StatusScoreDescriptionMitigations
7 TLS Downgrade Attack Tampering High Open 7.5 An active network attacker may attempt to downgrade TLS to an older, vulnerable version (e.g., TLS 1.0) to intercept traffic. Enforce TLS 1.2+ via server configuration. Enable HSTS with includeSubDomains and preload. Disable SSL 3.0, TLS 1.0, TLS 1.1 on all edge nodes.

OAuth2 PKCE Flow (Data Flow)

Description: PKCE-based OAuth2 authorization flow redirected via browser to the external IdP.

#TitleTypeSeverity StatusScoreDescriptionMitigations
8 Authorization Code Interception Information disclosure High Open 7.2 Without PKCE, a malicious app on the same device could intercept the authorization code from the redirect URI and exchange it for tokens. Enforce PKCE (S256 code challenge) for all public clients. Validate code_challenge on the authorization server. Ensure short-lived authorization codes (< 1 minute).

Checkout Service (Process)

Description: Orchestrates the checkout flow: validates cart, applies promotions, calls fraud detection, and triggers payment authorization via Stripe.

#TitleTypeSeverity StatusScoreDescriptionMitigations
24 Price Manipulation in Cart Validation Tampering Critical Open 9.2 If the Checkout Service trusts client-supplied prices rather than re-fetching from the Product Catalog, an attacker can manipulate the cart total and purchase items at arbitrary prices. Never trust client-supplied prices. Always re-fetch authoritative prices from the Product Catalog Service at checkout time. Validate total amount before payment authorization.
25 Promo Code Abuse Elevation of privilege Medium Open 5.1 Automated scripts can brute-force valid promotional codes, applying large discounts without authorization. Rate limit promo code validation per user/IP. Implement code complexity requirements. Log all failed validation attempts and alert on anomalies.
26 Double-Spend via Concurrent Checkout Requests Tampering High Open 7.6 Concurrent checkout requests for the same cart can cause double charges or overselling of limited-stock items due to TOCTOU race condition. Use pessimistic locking or distributed mutex (Redis SETNX) on cart during checkout. Implement idempotency keys for payment calls. Check inventory with SELECT FOR UPDATE.
27 Checkout Denial of Service Denial of service High Open 7.3 An attacker can initiate thousands of checkout sessions without completing payment, exhausting inventory locks and blocking legitimate purchases. Implement checkout session timeouts. Release locks after timeout. Require CAPTCHA for guest checkouts. Rate limit checkout initiations per account.

Payment Service (Process)

Description: Internal service managing payment method tokenization, charge lifecycle, refunds, and webhook processing from Stripe.

#TitleTypeSeverity StatusScoreDescriptionMitigations
28 Stripe Webhook Forgery Spoofing Critical Open 9.0 Forged webhook events claiming payment success could trigger order fulfilment without actual payment, resulting in financial loss. Validate Stripe-Signature header using HMAC-SHA256 with the webhook signing secret. Reject events that fail signature verification. Process webhooks idempotently.
29 Payment Token Exposure in Logs Information disclosure High Mitigated 7.0 Logging full request/response payloads may inadvertently log Stripe payment method IDs or tokens, which could be misused. Implemented structured logging with field-level masking for payment-related fields. Log redaction verified by automated test suite.
30 Refund Authorization Bypass Elevation of privilege Critical Open 8.9 Missing role-based access control on the refund endpoint could allow any authenticated user to initiate refunds for any order. Restrict refund endpoint to admin and customer-service roles. Validate that refund amount does not exceed original charge. Require 2FA for refunds above threshold.

Payment Token Vault (Store)

Description: Encrypted data store for Stripe payment method tokens (not raw card data). Tokens are tied to customer accounts for saved payment methods.

#TitleTypeSeverity StatusScoreDescriptionMitigations
31 Mass Token Exfiltration Information disclosure Critical Open 9.5 A compromised Payment Service with database access could bulk-export all payment tokens, enabling fraudulent charges across all stored payment methods. Enforce column-level encryption with keys stored in KMS. Implement DLP alerting on bulk SELECT queries. Restrict vault access to Payment Service service account only.
32 Token Database Tampering Tampering High Open 7.8 An attacker with write access could reassign payment tokens between customers, causing payments to be charged to wrong accounts. Cryptographically bind tokens to customer IDs using signed records. Enable database row-level security. Audit all UPDATE operations on the vault table.

Payment Audit Log (Store)

Description: Append-only write-once log of all payment events for compliance, dispute resolution, and forensic analysis. Stored in S3 with Object Lock.

#TitleTypeSeverity StatusScoreDescriptionMitigations
No threats

Payment Request (gRPC) (Data Flow)

Description: Checkout Service instructs Payment Service to charge the customer's saved payment method.

#TitleTypeSeverity StatusScoreDescriptionMitigations
33 Service Identity Spoofing Spoofing High Open 7.9 Without mutual TLS, a rogue service in the VPC could impersonate the Checkout Service and trigger unauthorized payments. Enforce mTLS with short-lived certificates issued by an internal CA (SPIFFE/SPIRE). Validate client certificate SANs on every call.

Stripe API (HTTPS) (Data Flow)

Description: Payment Service calls Stripe REST API to create charges, capture payments, and process refunds.

#TitleTypeSeverity StatusScoreDescriptionMitigations

Store Stripe API keys in a secrets manager (e.g., AWS Secrets Manager). Rotate keys regularly. Monitor for unauthorized API usage via Stripe radar and webhooks.
